## 1. ファイル管理ルール

### 絶対守ること

- **M5Stickのスケッチ（`.ino`）とゲームコード（`.cpp`）は必ず同じリポジトリで管理する。**  
  → スケッチがなければ、センサー軸のマッピングを後から確認できなくなる（旧プロジェクトで発生した問題）。

- **旧プロジェクト（`授業版`）には手を加えない。**  
  → アーカイブとして保持。Camagame_rider は独立した複製として進める。

- **アセット（PNG/WAV）は `cgprog/assets/` にまとめる。**  
  → `.cpp` ファイルと同じ階層に混在させない。

### ディレクトリ命名規則

```
Camagame_rider/
├── arduino/         ← Arduinoスケッチのみ
├── cgprog/          ← C++コード（ゲーム本体）
│   └── assets/      ← 画像・音声
├── LiDAR/           ← LiDAR関連（バイナリ・設定・共有ファイル）
└── docs/            ← 設計メモ・ドキュメント
```

## 2. コーディングルール（C++）

### 関数の役割分離（必須）

`timer()` のような万能関数を作らない。以下の役割ごとに関数を分ける：

| 関数名 | 役割 |
|--------|------|
| `updateInput()` | M5Stickのシリアルデータ読み込みと解析 |
| `updateLiDAR()` | footpoint.txt の読み込みと足位置更新 |
| `updatePlayer()` | プレイヤー角度・防御判定 |
| `updateEnemies()` | 敵の移動・攻撃AI |
| `updateBullets()` | 弾の移動・衝突判定 |
| `updateWave()` | WAVE管理（進行条件の判定） |
| `drawFloor()` | 床面ウィンドウの描画 |
| `drawWall()` | 壁面ウィンドウの描画 |

### グローバル変数の扱い

- 現状はグローバル変数で管理を続ける（大きな構造変更は後回し）
- ただし **新しくグローバル変数を追加するときは必ず `README.md` の変数一覧を更新する**

### ブレース（`{}`）の省略禁止

```cpp
// NG
if (condition)
    doSomething();

// OK
if (condition) {
    doSomething();
}
```

→ `{}` を省略すると、後から行を追加したときに条件の外に出てしまう事故が起きやすい。  
（※旧プロジェクトの防御バグは当初これが原因と誤診したが、実際は「防御判定の二重定義」が原因だった。  
それでもブレース省略は事故の温床なので、習慣として省略しない。）

### コメントルール

- 変数のコメントは **「何のための値か」** を書く（「何の値か」ではなく）
- `// ★` や `// ▼▼▼` のマーカーは使わない（後から検索しにくい）
- TODOは `// TODO(優先度):` 形式で書く
  - 例: `// TODO(高): 軸確認後にval[]のマッピングを確定させる`

## 3. センサー軸ルール（M5StickC）

### val[] インデックスとセンサー軸の対応

以下は **M5StickCをLong辺が上（画面が正面）の状態で送信した場合** の対応：

| val[] | センサー変数 | 物理的な意味 | 鎌での役割 |
|-------|-------------|------------|-----------|
| val[0] | acc_x | 左右傾き（G単位） | 未使用（防御判定の予備） |
| val[1] | acc_y | 前後傾き（G単位） | 未使用 |
| val[2] | acc_z | 重力方向（G単位） | 未使用 |
| val[3] | gyro_x | Pitch（前後回転の角速度 °/s） | **縦振り攻撃** |
| val[4] | gyro_y | Roll（左右回転の角速度 °/s） | **横振り攻撃** |
| val[5] | gyro_z | Yaw（水平回転の角速度 °/s） | **プレイヤー向き回転** |

> ⚠️ **取り付け向きを変えたらこの表を必ず更新すること**  
> M5Stickを横向きに固定した場合は gyro_x と gyro_y が入れ替わる。

### 軸確認の手順（毎回必須）

1. ゲームを `d` キーのデバッグモードで起動
2. M5Stickを**振らずに静止**させて、val[3]～val[5]が概ね0に近いことを確認
3. **縦振り（上から下）** をして val[3] が大きく動くことを確認
4. **横振り（左から右）** をして val[4] が大きく動くことを確認
5. **水平回転（時計回り）** をして val[5] の符号を確認
6. 確認後、閾値（`atk_thresh`）を実際の値に合わせて設定

### シリアル送信フォーマット

```
acc_x,acc_y,acc_z,gyro_x,gyro_y,gyro_z\n
```

カンマ区切り6値、末尾改行。  
送信間隔は **20ms（50Hz）** に統一する（ゲームのfr=50に合わせる）。

## 4. LiDAR運用ルール

### 起動順序（必ず守ること）

```bash
# 1. まずLiDARアプリを起動（設置環境専用・本リポジトリには非同梱）
./（LiDARアプリ）

# 2. footpoint.txt が更新されていることを確認してからゲームを起動
cat footpoint.txt   # 0 以外の数値があればOK

# 3. ゲームを起動
cd ../cgprog
./game
```

LiDARが動いていないままゲームを起動すると `footNum=0` のままになり、プレイヤーが動かない。

### footpoint.txt フォーマット

```
<検出点数N>
x0,y0
x1,y1
...
xN-1,yN-1
```

座標系はゲーム画面座標（-550 〜 550 / 0 〜 700）に変換済みであること。

### scanarea.txt フォーマット

```
<width>,<height>,<flip_flag>
```

例: `<width>,<height>,<flip>` → 幅・高さ・反転フラグ

## 5. 2画面構想ルール

### 設計の基本方針

**床面がゲームのメイン画面**。プレイヤーは床を見ながら歩き回って戦う。  
壁面はスコアなどの情報表示だけ。

```
[プレイヤー視点]

  ↓ 足元（床）を見る
    敵・ボス・弾・攻撃エフェクト・プレイヤー位置 が全部ここに映る
    LiDARで足を追跡してプレイヤーを動かす

  ↑ 顔を上げて壁を見る
    スコア・HP・残り時間・WAVE数 だけが表示される
```

### 床面ウィンドウ（Window ID: 0）— メイン

- 解像度: 1100×700（旧コードから継承）
- **表示内容（ゲームの全要素）**:
  - プレイヤーキャラクター（LiDARの足位置に連動）
  - 歩いた軌跡（footprint）
  - 敵・ボスキャラクター
  - 敵の弾・反射弾
  - 縦振り・横振り・防御のエフェクト
  - アイテム
  - 背景画像
  - タイトル・ルール・クリア・ゲームオーバー画面（床専用画像）
- 担当関数: `drawFloor()`

### 壁面ウィンドウ（Window ID: 1）— UIのみ

- 解像度: 1100×700
- **表示内容（情報表示のみ）**:
  - スコア
  - プレイヤーHP
  - 残り時間（WAVE1）
  - WAVE数・残り敵数
- 担当関数: `drawWall()`

> **将来的な拡張案（現時点では未実装）**:  
> 壁面にも敵を出現させて、壁に向かって攻撃できるモードを追加する。  
> まずは床のゲームを完成させてから検討する。

### 2画面の同期ルール
- 両ウィンドウは **同一プロセス内** で `glutSetWindow()` で切り替えて描画する
- `glutMainLoop()` は1回のみ呼ぶ
- 壁面が閉じられてもゲーム（床面）は続行できること

## 6. ゲームルール（正式版）

### WAVEの設計意図

| WAVE | 目的 | 敵の弾 | 設計意図 |
|------|------|--------|---------|
| WAVE 1 | 30秒間でできるだけ多く倒す | **なし** | チュートリアル。鎌の縦振り・横振りを練習する場。スコア稼ぎ。 |
| WAVE 2 | 雑魚を合計4体倒す | **あり** | 弾を防御で跳ね返しながら戦う練習。反射攻撃を覚える。 |
| WAVE 3 | ボスを倒す | **あり（3パターン）** | 縦振り・横振り・反射をフル活用して戦う。 |

> WAVE1で敵が弾を撃たないのはコードでも実装済み（`currentWave >= 2` の条件がある）。

### 操作の役割

| 操作 | 機能 | 備考 |
|------|------|------|
| 歩く | プレイヤーが移動する | LiDARが足を追跡 |
| 縦振り | 強攻撃（前方・射程長い） | ダメージ80 |
| 横振り | 速攻（周囲・連打向き） | ダメージ20 |
| 静止（防御） | 弾を跳ね返す | 前方45度以内の弾を反射できる |

### ベースファイル

新プロジェクトは旧 `main.cpp`（無印・最新・最終版・2画面・発表で動いた版）をベースに再構築する。  
→ `Camagame_rider/cgprog/main.cpp` に複製済み。

### 操作設計の確定事項（2026-06-22 確認済み）

**向きの決め方** … 歩いた方向に自動で向く（LiDARの移動方向が主軸）
- **タレット旋回**（ジャイロZ`val[5]`での手動回転）は**外す**（向きの二重制御 BUG-008 を解消）
- **オートエイム**（攻撃した瞬間、200以内の敵にスナップ）は**残す**（歩行向きと両立・攻撃が当てやすい）
- 静止中は直前の向きを保持する
- 向きが効くのは「縦振りの方向」と「防御の反射方向」
- → 最終的に向きは「歩行（普段）＋オートエイム（攻撃時）」の2系統に整理される

**防御（ガード）の出し方** … その場で立ち止まると発動（LiDARの静止のみ）
- M5Stick静止との二重判定は**やめる**（BUG-001 を解消し、判定を1か所に統一）
- LiDARノイズに強い静止判定にする（1フレーム差分でなく数フレームの平均移動量で判定）

### 防御（反射）の仕組み

- その場で立ち止まる（LiDARが静止）→ 防御状態
- 防御中に弾がプレイヤー前方45度以内から来た → 反射弾になる（速度1.5倍）
- 反射弾が敵に当たる → 50ダメージ
- 立ち止まる＝歩いた方向を向いたまま → その方向に弾を跳ね返せる（将来の「壁の敵を反射で倒す」に繋がる）

## 6-3. 将来の拡張アイデア（未実装）

### 壁面の敵（空を飛ぶ敵）

- **設定**: 壁に映る敵は「空中に浮いている」ため、鎌が直接届かない
- **倒し方**: 床の敵が撃ってきた弾をガードして跳ね返す → 跳ね返した弾が壁面の敵に当たって倒せる
- **ゲームデザイン上の意義**:
  - 床の敵 → 縦振り・横振りで直接倒す
  - 壁の敵 → 反射でしか倒せない（防御が攻撃手段になる）
  - 2画面を使う理由が明確になる
- **実装前提**: まず床のゲームを完成させてから着手する
